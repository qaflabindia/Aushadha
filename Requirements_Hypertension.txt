Functional and Technical Requirements Specifications Problem Statement-3 AYUSH Hypertension Proof of Concept (For delivery use by Team and Developers  )

1 Problem statement and Stage 1 scope
Problem Statement 3 AI enabled Integrated Early Warning, Treatment and Lifestyle Recommendation System The AHMIS ecosystem generates large scale clinical and care delivery data across AYUSH centres. The goal is to use AI to improve structured Electronic Health Records creation, support early warning signals for localised surges, and enable personalised, evidence informed AYUSH aligned treatment and lifestyle recommendations with explainability and clinical trust. 
Stage 1 scope statement for developers
Stage 1 Proof of Concept for hypertension with three use cases. Use case 1 multilingual clinician note to structured EHR in Hindi and English with clinician review. Use case 2 personalised AYUSH aligned care plan for hypertension with Clinician Trust Layer and safety gates. Use case 3 early warning review signals from aggregated synthetic counts with alert review queue. Stage 1 uses synthetic and public data only and does not claim live AHMIS integration. 
2 Authentication, roles, and sessions
2.1 Authentication for PoC
PoC shall support local demo accounts stored in a database table.
Minimum demo accounts: one Clinician, one Public health reviewer, one Administrator.
Passwords must be hashed using bcrypt or Argon2.
Password reset for PoC: Administrator can set a new password for a user.
Admin bootstrap: first run creates an admin account from environment variables.
2.2 Session management
Session timeout after 30 minutes of inactivity.
Forced logout after 8 hours of continuous session.
Unauthorized action returns 403 and generates an audit event with result DENY.

5 Extraction rules to prevent hallucination
Do not invent BP values. Extract BP only if explicitly present in the note or entered by clinician.
If multiple BP readings exist, select the last mentioned reading in the note.
If BP values conflict or are out of range, set BP to Unknown and add warning text for clinician review.
If symptoms are unclear, set symptom flags to Unknown, not False.
pregnancy_suspected is set True only if pregnancy is explicitly stated; otherwise Unknown.
6 Confidence derivation rules

Medium

Low

Item must be safety Caution and must include review prompt. Low confidence cannot be Safe.

7.3 UC3 Early warning alerts
7.3.1 Input metric definitions
crisis_suspected_count is count of visits in week where bp_category is CrisisSuspected.
stage2_count is count of visits in week where bp_category is Stage2.
urgent_symptom_count is count of visits in week where symptom_chest_pain is True OR symptom_breathlessness is True OR symptom_neurological is True.
total_visits is total number of hypertension related visits captured for that locality and week.
7.3.2 Deviation score definition
Compute rate_per_100 = (metric_count / total_visits) * 100 for crisis_suspected_count, urgent_symptom_count, and stage2_count.
Compute baseline mean and baseline standard deviation on rate_per_100 over the previous 8 weeks.
Compute z_score = (current_rate_per_100 - baseline_mean) / max(baseline_std, 0.5) where 0.5 is in rate points per 100 visits.
Compute deviation_score = max(z_crisis, z_urgent, 0.5 * z_stage2).
Eligibility gate: total_visits must be at least 50 AND at least one metric count must be at least 5.
Persistence: deviation_score must be at least 2.0 for 2 consecutive eligible weeks. Missing week or ineligible week breaks persistence.

8 Prohibited outputs and guardrail enforcement
8.1 Prohibited outputs list
Any cure claim such as cure, permanently fix, guaranteed
Any dosing instruction for herbs or medicines including mg, grams, capsules, tablets, drops, times per day
Any instruction to stop, reduce, or change prescribed medicine
Any emergency management steps beyond referral guidance
Any breath retention instruction, strong hyperventilation, or high intensity routines in red flag cases
8.2 Guardrail enforcement behaviour
When guardrail triggers, the system must not show the offending text as a recommendation.
For the clinician UI, replace the item with a placeholder: Blocked by safety rule with the reason category and the item_id.
For exports, blocked items are omitted from plan_sections and recorded only in a separate guardrail_log object not shown as recommendations.
A guardrail violation writes an audit event with action GUARDRAIL_BLOCK, result BLOCKED, and a reason_code such as GUARDRAIL_DOSING or GUARDRAIL_MED_CHANGE.
Guardrail unit tests must exist for each prohibited class.
